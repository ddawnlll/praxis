# Report Model — ACCP and Runtime Reports

> This document defines the report model for PRAXIS v0.1. Two report types exist: ACCP reports (design-phase lock reports, audits) and Runtime reports (per-attempt execution results).

## Report Types

### ACCP Reports

ACCP (Artifact Chain of Custody Protocol) reports are used for:
- Schema pack locks (D3)
- Phase completion verification (P1, P2)
- Design audits and scorecards
- Configuration validation

**Format:** YAML (`.accp.yaml`) + optional Markdown summary (`.summary.md`)

**Location:** `reports/accp/`

### Runtime Reports

Runtime reports are generated by the Truth Kernel after pipeline execution:
- Per-attempt gate verdict reports
- Final completion reports
- Repair reports
- Attempt history

**Format:** YAML (`.report.yaml`) + Markdown summary (`.summary.md`)

**Location:** `.praxis/runs/<run-id>/` and `.praxis/reports/`

## ACCP Report Model

### Structure

```yaml
# ACCP Report — <title>
# accp_version: 2.0.0
# source_format: ACCP-YAML
# id: <unique-report-id>
# title: <report-title>
# created_at: <iso-timestamp>
# execution_mode: <mode>
# analyzer: <analyzer-name>

report:
  id: <short-id>
  status: PASS | HOLD | FAIL
  status_meaning: >-
    Human-readable summary of the report status.

executive_summary: |
  Markdown-formatted executive summary.
  Multiple lines.

# Report-specific sections follow below.
# Common sections:
# - source_reconciliation (for schema/design reports)
# - git_state (for state audits)
# - file_inventory_summary (for audits)
# - risks_or_missing_items (for audits)
# - scorecard (for design scorecards)

acceptance_criteria:
  - id: "AC-001"
    description: "..."
    status: "PASS" | "FAIL"
    evidence: "..."

completion_gate:
  done_when:
    - "...: ✓"
    - "...: ✓"
  status: "DONE" | "INCOMPLETE"

exact_commands_run:
  - "command1"
  - "command2"
```

### Required Headers

Every ACCP report MUST start with YAML comment headers:

| Field | Description | Required |
|-------|-------------|----------|
| `accp_version` | ACCP protocol version | Always `2.0.0` |
| `source_format` | Always `ACCP-YAML` | Yes |
| `id` | Unique report identifier (SCREAMING_SNAKE) | Yes |
| `title` | Human-readable report title | Yes |
| `created_at` | ISO 8601 timestamp | Yes |
| `execution_mode` | What mode produced this report | Yes |
| `analyzer` | Who/what produced it | Yes |

### Common Sections

| Section | Purpose | Required |
|---------|---------|----------|
| `report.id` | Short report ID | Yes |
| `report.status` | PASS/HOLD/FAIL | Yes |
| `report.status_meaning` | Explanation of status | Yes |
| `executive_summary` | Markdown summary | Yes |
| `acceptance_criteria` | AC pass/fail list | No (audit reports) |
| `completion_gate` | Gate status | No (audit reports) |
| `exact_commands_run` | Commands used during report generation | No (audit reports) |

## Runtime Report Model

### Structure

```yaml
# PRAXIS Runtime Report
# type: run_report | repair_report | completion_report
# report_version: 0.1.0

report:
  id: "run-<uuid>"
  type: "run_report"
  planId: "<plan-id>"
  attemptId: "<attempt-id>"
  startedAt: "<iso-timestamp>"
  completedAt: "<iso-timestamp>"
  overallVerdict: "PASS" | "HOLD" | "FAIL"

pipeline:
  gates:
    - gateName: "SchemaGate"
      verdict: "PASS"
      reasonCodes: ["SCHEMA_PASS"]
      durationMs: 45
    - gateName: "LockGate"
      verdict: "PASS"
      reasonCodes: ["LOCK_PASS"]
      durationMs: 12
    - gateName: "EvidenceGate"
      verdict: "PASS"
      reasonCodes: ["EVIDENCE_PASS"]
      evidenceCount: 42
      durationMs: 88
    - gateName: "WiringGate"
      verdict: "PASS"
      reasonCodes: ["WIRING_PASS"]
      unitsChecked: 7
      durationMs: 65
    - gateName: "ExecGate"
      verdict: "HOLD"
      reasonCodes: ["EXIT_CODE_NONZERO", "NO_TESTS_FOUND"]
      commandsRun: 3
      commandsPassed: 2
      commandsFailed: 1
      durationMs: 15420
    - gateName: "FinalGate"
      verdict: "HOLD"
      reasonCodes: ["CRITERIA_PARTIAL"]
      totalCriteria: 5
      passedCriteria: 4
      failedCriteria: 1
      advisoryCriteria: 0
      durationMs: 23

criteria:
  - id: "AC-001"
    taskId: "task-1"
    description: "Auth module exists"
    verdict: "PASS"
    verificationType: "file_exists"
    deterministicEvidence: true
    evidenceRefs: ["evt-001"]
  - id: "AC-002"
    taskId: "task-1"
    description: "All tests pass"
    verdict: "HOLD"
    verificationType: "test_output"
    deterministicEvidence: true
    evidenceRefs: ["evt-005"]
    detail: "2/5 tests failed"

commands:
  - commandId: "CMD-001"
    command: "bun test"
    verdict: "HOLD"
    exitCode: 1
    durationMs: 15000
    timedOut: false
    evidenceRef: "evt-005"

repair:
  repairPacketGenerated: true
  repairPacketPath: ".praxis/repairs/rp-001.json"
  strategy: "initial"
  priorFailureCount: 0

evidence:
  totalRecords: 42
  ledgerPath: ".praxis/runs/run-001/evidence.jsonl"
  chainIntegrity: "CLEAN"

summary: |
  ## Execution Summary
  
  **Plan:** plan-20260620-001
  **Attempt:** attempt-01
  **Verdict:** HOLD
  
  **Gates:**
  - SchemaGate: ✅ PASS
  - LockGate: ✅ PASS
  - EvidenceGate: ✅ PASS (42 evidence records)
  - WiringGate: ✅ PASS (7 units checked)
  - ExecGate: ⚠️ HOLD (2/3 commands passed)
  - FinalGate: ⚠️ HOLD (4/5 criteria passed)
  
  **Failed Criteria:**
  - AC-002: 2/5 tests failed (test_output)
  
  **Repair:** RepairPacket generated at .praxis/repairs/rp-001.json
  **Strategy:** initial (attempt 1)
```

### Runtime Report Types

| Type | Generated When | Purpose |
|------|---------------|---------|
| `run_report` | Every pipeline execution | Full gate-by-gate results |
| `repair_report` | When RepairPacket generated | Focus on failures and repair strategy |
| `completion_report` | FinalGate PASS | Success summary |

### Required Sections Per Report Type

| Section | run_report | repair_report | completion_report |
|---------|-----------|---------------|-------------------|
| `report` | Required | Required | Required |
| `pipeline` | Required | Required | Required |
| `criteria` | Required | Failed only | Required |
| `commands` | Required | Required | Required |
| `repair` | Required | Required | Skipped |
| `evidence` | Required | Required | Required |
| `summary` | Required | Required | Required |

## Summary Markdown Format

Each YAML report SHOULD have a companion `.summary.md` file for human reading. The summary is generated from the YAML data and contains:

```markdown
# <Report Title>

**Generated:** <timestamp>
**Report:** <path to .report.yaml>

## Status

**Overall:** PASS / HOLD / FAIL

## Gate Results

| Gate | Verdict | Duration |
|------|---------|----------|
| SchemaGate | ✅ PASS | 45ms |
| ...

## Criterion Results

| ID | Description | Verdict |
|----|------------|---------|
| AC-001 | Auth module exists | ✅ PASS |

## Commands

| Command | Verdict | Exit Code | Duration |
|---------|---------|-----------|----------|
| bun test | ⚠️ HOLD | 1 | 15s |

## Evidence

- **Total records:** 42
- **Chain integrity:** CLEAN

## Repair

- **Packet generated:** rp-001.json
- **Strategy:** initial
- **Attempt:** 1/10
```

## Report Directory Structure

```
.praxis/
  runs/
    <run-id>/
      evidence.jsonl         ← Evidence ledger
      diffs/                 ← Diff content files
      command-output/        ← Command output files
      test-output/           ← Test result files
      snapshots/             ← Filesystem snapshots
      run.report.yaml        ← Runtime report
      run.summary.md         ← Runtime report summary
  
  repairs/
    <repair-id>.json          ← RepairPacket
  
  reports/
    <report-id>.report.yaml  ← Standalone reports
    <report-id>.summary.md   ← Standalone summaries

reports/
  accp/
    *.accp.yaml               ← ACCP design/audit reports
    *.summary.md              ← ACCP summaries
```

## Report Versioning

Reports are NOT versioned per-file. Instead:
- Each report has a `created_at` timestamp
- The report format version is in the header comment
- Reports are append-only — never overwritten
- New attempts create new reports with new IDs

## Design Decisions

| Decision | Answer | Rationale |
|----------|--------|-----------|
| Dual format (YAML+MD)? | Yes | YAML for machines, MD for humans. Both generated from same data. |
| ACCP YAML for runtime? | No | Runtime reports use `.report.yaml` suffix to distinguish from ACCP design reports |
| Include full evidence in report? | No | Report references evidence via IDs and paths. Full content stays in ledger. |
| Include diagnostic details? | Yes | Every gate verdict includes reason codes and diagnostics |
| Summary inline in YAML? | Yes | The `summary` field in the YAML report is human-readable markdown. Separate `.summary.md` file is extracted from it. |
